TR-Graph: Task Command Repair for Safe LLM-Controlled Robot Systems

A graph-structured framework for transforming defective task commands into safe, executable alternatives that maintain fidelity to the original command.

Anonymous Authors

Paper Video Results

Abstract

LLM-controlled robot systems often receive natural language commands that are underspecified, infeasible, or violate safety preconditions. Current systems either reject such commands entirely or execute them blindly, risking physical harm. We propose TR-Graph, a graph-structured framework for transforming defective task commands into safe and executable alternatives that maintain fidelity to the original command. TR-Graphs draws on multiple safety standards for its framing, including ISO 12100 and ISO/TS15066. We evaluate TR-Graph on external benchmarks, real-world robot experiments with Franka Research 3 Manipulator, and qualitative mechanistic analysis. Across evaluations, TR-Graph produced safer and more executable repairs than all baselines.

Method

TR-Graph Pipeline

LLM-controlled robot systems often receive natural language commands that are underspecified, infeasible, or incompatible with grounded scene context and robot capabilities, and current systems either reject such commands entirely or execute them blindly, risking physical harm. TR-Graph intercepts these commands, builds a task repair graph linking grounded scene entities, task variables, and violation pathways, and generates repaired candidates guided by six formally defined repair operators.

1

Stage I: Grounded Task Decomposition

Given a natural-language command, TR-Graph performs a frame decomposition to determine the core frame of the task, which arguments are central, and what roles they play. The command is then parsed into a structured task specification τ = ⟨a, O, L, r, P, C, M, H⟩, capturing action, objects, locations, recipient, parameters, constraints, manner, and horizon. This specification is aligned to the grounded scene graph through a grounding map that resolves object references, location references, and recipients to scene-graph nodes. Grounding may produce one of four outcomes: resolved, ambiguous, infeasible, or skill-infeasible. Safety-critical ambiguities are resolved by posing clarification questions to the user rather than guessing.

2

Stage II: Graph-Guided Diagnosis

The grounded task is evaluated prior to execution to determine whether it is safe, feasible, and compatible with the current environment and robot skill set. If defective, the diagnosis produces defect type, hazard type, severity, probability, causal variables, causal edges, and available alternatives. TR-Graph then constructs a directed task-conditioned repair graph Gτ = (Vτ, Eτ, Sτ), where nodes represent task variables, grounded resources, and defect-relevant entities, and directed edges encode dependency and propagation relations. Violation sink nodes represent unsafe or infeasible outcomes. The system traces directed paths from task-variable nodes toward violation sinks to identify which assignments are causally responsible for the defect. Nodes on short or high-confidence paths to sinks are treated as repair-relevant.

3

Stage III: Graph-Guided Repair Synthesis and Validation

Using the diagnosis, graph, and operator priorities from a contextual bandit (LinUCB), TR-Graph generates repair candidates. Each candidate is represented as a structured repair package R(τ′) = ⟨desc, π, Kτ′, gτ′, Δ(τ, τ′), Ωτ′⟩, exposing the repaired description, action sequence, required skills, expected end state, changed variables, and applied operators. Candidates pass through a gated validation pipeline: safety re-evaluation, graph consistency checking, operator precondition validation, and capability/executability checking. Only repairs that are safe, structurally coherent, operator-valid, and semantically faithful survive to ranking.

4

Stage IV: Execution and Online Adaptation

Validated candidates are ranked by a scoring objective that balances task fidelity (affinity) against graph-weighted intervention cost, favoring repairs that minimize unnecessary deviation from the original command. TR-Graph produces a user-facing explanation summarizing how each repair option changed the original task graph so that the path to the violation sink was broken. Repaired candidates are presented as recommendations and executed only with user approval. After execution, the contextual bandit is updated using the graph context and the winning repair, helping the system learn operator prioritization over time.

Repair Operators

TR-Graph repairs commands by applying a small set of formally defined intervention operators inspired by the ISO 12100 safety standard. These operators define the permitted modes of intervention over the task representation:

1

Substitute

Replace a task argument with a safer or feasible alternative sharing required affordances.

2

Constrain

Add an explicit safety, feasibility, or contextual constraint as a pre- or post-condition.

3

Retarget

Redirect the task toward a different grounded object, person, or location in the scene.

4

Reorder

Change the order of subtasks when the current sequencing contributes to risk.

5

Relax

Weaken or remove a problematic requirement when no repair satisfies it fully.

6

Decompose

Split a compound command into simpler, individually executable substeps.
